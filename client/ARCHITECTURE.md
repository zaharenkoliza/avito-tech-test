# Архитектура клиента (FSD)

## Цели
- Держать бизнес-логику явной и легко находимой по слоям.
- Изолировать переиспользуемую доменную логику в `entities`, а пользовательские сценарии - в `features`.
- Держать `pages` "тонкими": только оркестрация, без низкоуровневых UI-деталей.

## Слоистая структура

```text
src/
  app/                  # инициализация, роутер, store, глобальные границы
  pages/                # оркестрация на уровне маршрутов
  widgets/              # составные UI-блоки для страниц
  features/             # действия/сценарии пользователя + состояние фич
    ads-list-filters/
    theme-toggle/
  entities/             # бизнес-сущности и доменные хелперы
    ad/
  shared/               # общая инфраструктура (api, config, utils, storage)
  test/                 # настройка тестов
```

## Текущие слайсы

1. `entities/ad`
- Типы и метки объявлений (`Category`, `Item`, `ItemWithRevision`, `CATEGORY_LABELS`)
- Доменный хелпер объявления `getMissingFields`

2. `features/ads-list-filters`
- Состояние фильтров списка (`ListState`, actions/reducer слайса)
- Маппинг URL query <-> state (`buildListQueryFromSearchParams`, `buildSearchParamsFromListState`)
- Опции категорий (`ALL_CATEGORIES`)

3. `features/theme-toggle`
- Глобальное состояние темы и его персистентность

4. `widgets/*`
- `ads-list`: блоки фильтров/результатов/тулбара
- `ad-details`: блоки заголовка/галереи/характеристик/ревизий
- `ad-edit`: блоки формы/чата/ассистента

## Правила зависимостей

1. Разрешено только направление сверху вниз:
- `app -> pages -> widgets -> features -> entities -> shared`

2. Импорты между слайсами:
- Использовать публичные API (`index.ts`) каждого слайса
- Избегать deep import во внутренние файлы другого слайса

3. Размещение состояния:
- Расчеты сущностей/домена - в `entities`
- Состояние пользовательских потоков - в `features`
- Локальное состояние UI страницы - в `pages` или `widgets`

## Фокус тестирования

1. `entities/*/lib`: доменные правила и крайние случаи.
2. `features/*/model|lib`: reducer и преобразования query/state.
3. `pages/*`: покрытие сценариев с мокированным backend.
4. `shared/*`: инфраструктурные утилиты и сериализация.
