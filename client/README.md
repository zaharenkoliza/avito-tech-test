# Avito Tech Test: Client

Frontend-часть проекта на React + TypeScript + Vite.

## Запуск

Требования:
- Node.js 20+
- npm 10+
- запущенный backend на `http://localhost:8080`

Команды:

```bash
cd client
npm install
npm run dev
```

Клиент будет доступен на `http://localhost:5173`.

Если API поднят на другом адресе:

```bash
VITE_API_BASE_URL=http://localhost:8080
```

## Скрипты

```bash
cd client
npm run dev
npm run build
npm run lint
npm run lint:fix
npm run format
npm run format:check
```

## Архитектура

Клиент организован по слоям:

```text
src/
  app/                  # entrypoint, router, store, providers
  pages/                # orchestration на уровне маршрутов
  widgets/              # крупные блоки страниц
  features/             # пользовательские сценарии и глобальные фичи
  entities/             # доменные сущности и хелперы
  shared/               # api, storage, utils, config
```

Текущая раскладка ответственности:
- `features/ads-list-filters` хранит query-state списка и синхронизацию с URL;
- `features/theme-toggle` хранит глобальное состояние темы;
- `entities/ad` содержит типы объявления, метки категорий и доменные хелперы;
- `widgets/ads-list`, `widgets/ad-details`, `widgets/ad-edit` собирают UI страниц из более мелких частей.

Маршрутизация остается простой и лежит в `app/router.tsx`:
- `/ads`
- `/ads/:id`
- `/ads/:id/edit`
- все остальные маршруты редиректятся на `/ads`

### Устойчивость интерфейса

В `app/ErrorBoundary.tsx` добавлен глобальный `ErrorBoundary`, которым обернут роутер приложения.

Его задача - перехватывать непредвиденные runtime-ошибки в дереве React-компонентов и вместо полного падения интерфейса показывать fallback-состояние с сообщением об ошибке и кнопкой перезагрузки страницы.

Это решение полезно, так как пользователь не остается с "белым экраном", если одна из частей UI упала при рендере.

### Роль кастомных хуков

Заметная часть архитектуры собрана вокруг кастомных хуков, чтобы страницы не разрастались и не смешивали UI с orchestration-логикой.

- `useAdsListPageState`
  инкапсулирует состояние страницы списка: гидрацию из URL, debounce поиска, загрузку данных, синхронизацию query-параметров и вычисляемые значения вроде `totalPages`.
- `useAdDetailsPageState`
  отвечает за загрузку объявления, обработку невалидного `id`, вычисление незаполненных полей, подготовку характеристик и флага редактирования.
- `useAdAi`
  изолирует всю AI-логику страницы редактирования: генерацию описания, подсказку цены, чат, состояния загрузки, ошибки и отмену запросов.
- `useAdDraft`
  инкапсулирует работу с черновиками: чтение, сохранение и очистку данных в `localStorage`.
- `useAppDispatch` и `useAppSelector`
  типизированные обертки над Redux hooks, чтобы не размазывать типы по компонентам.

По сути, кастомные хуки используются как слой page-model:
- страницы собирают экран;
- хуки управляют состоянием, побочными эффектами и интеграцией с API;
- widgets получают уже подготовленные данные и callbacks.

Правила:
- `pages` остаются композиционными контейнерами;
- локальное поведение остается рядом со страницей или виджетом;
- доменные правила лежат в `entities`, а не в UI;
- зависимости идут сверху вниз: `app -> pages -> widgets -> features -> entities -> shared`;
- между слайсами предпочтительны импорты через публичные `index.ts`.

### Что где хранится

- Redux используется для
  filters/sort/page списка и темы.
- Локальное состояние страниц хранится в кастомных хуках:
  данные объявления, состояния загрузки, ошибки, AI-ответы, история чата, черновики.
- Доменная логика вынесена в `entities/ad`:
  типы, санитизация данных и определение незаполненных полей.
- Работа с API централизована в `shared/api/services.ts`, поэтому UI не знает деталей HTTP-запросов.
